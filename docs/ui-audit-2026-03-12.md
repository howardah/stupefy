# UI Audit Report

Date: 2026-03-12
Scope: lobby, room join/create, waiting room, gameplay board
Method: source audit against the `audit` skill criteria and the `frontend-design` anti-pattern rules
Verification gap: `nuxt` was not on `PATH`, and `npm exec nuxi typecheck` did not complete within the available window, so this report is based on source inspection rather than a completed typecheck run.

## Anti-Patterns Verdict

Verdict: Fail

This does not read like the worst 2024-2025 AI output, because the palette is warmer and less generic than the usual cyan-purple gradient stack. It still trips several frontend-design anti-patterns:

- repeated frosted rounded cards as the default container pattern
- a five-card gameplay metrics strip that reads like a dashboard template
- user-facing copy that repeatedly explains migration status and implementation details instead of gameplay
- generic serif fallback typography (`Georgia`, `Times New Roman`) instead of an intentional type system
- decorative blur used as a default panel treatment rather than a purposeful visual device

The result is coherent but still visibly "generated" in structure: safe card stacks, status badges, and development-state prose dominate the product experience.

## Detailed Findings by Severity

### Critical Issues

1. Waiting-room chat composer is missing a programmatic label, and the send button is icon-only.
   - Location: `components/WaitingRoomPanel.vue:156`, `components/WaitingRoomPanel.vue:166`
   - Severity: Critical
   - Category: Accessibility
   - Description: The chat input relies on placeholder text (`"Send a message"`) instead of a label, and the adjacent send action is an unlabeled icon-only button.
   - Impact: Screen-reader users cannot reliably identify the field purpose or the send control, which blocks a core room interaction.
   - WCAG/Standard: WCAG 1.3.1, 3.3.2, 4.1.2
   - Recommendation: Add a visible or screen-reader-only label for the input and an explicit accessible name for the send button.
   - Suggested command: `/harden`

2. Gameplay overlays are custom full-screen layers without dialog semantics, focus trapping, or modal focus management.
   - Location: `components/gameplay/GameplayAction.vue:16`, `components/gameplay/GameplayChooseCharacter.vue:29`
   - Severity: Critical
   - Category: Accessibility
   - Description: Both overlays are plain fixed-position containers. They do not declare `role="dialog"` / `aria-modal`, do not establish an accessible title relationship, and do not appear to trap focus or restore focus on close.
   - Impact: Keyboard and assistive-technology users can lose context, tab into obscured UI, or fail to understand that gameplay is blocked behind a modal state.
   - WCAG/Standard: WCAG 1.3.1, 2.1.2, 2.4.3, 4.1.2
   - Recommendation: Replace these layers with an accessible modal primitive or add proper dialog semantics, initial focus, focus trap, escape handling, and focus return.
   - Suggested command: `/harden`

### High-Severity Issues

1. The waiting-room refresh action is icon-only and has no explicit accessible name.
   - Location: `components/WaitingRoomPanel.vue:58`
   - Severity: High
   - Category: Accessibility
   - Description: The refresh button uses only an icon and loading state.
   - Impact: Screen-reader users may encounter an unnamed button in a core room-status surface.
   - WCAG/Standard: WCAG 4.1.2
   - Recommendation: Add a `label` or `aria-label`, and preserve that label during loading.
   - Suggested command: `/harden`

2. Card and character surfaces use CSS background images instead of semantic images.
   - Location: `components/gameplay/GameplayCard.vue:40`, `components/gameplay/GameplayCharacter.vue:33`, `components/gameplay/GameplayCharacterCard.vue:15`
   - Severity: High
   - Category: Accessibility / Performance
   - Description: Artwork is injected through inline `backgroundImage` styles on buttons. That prevents alt text, responsive image behavior, and explicit loading strategy.
   - Impact: Non-visual users lose descriptive media context, and visual users pay for heavier rendering and less controlled image loading.
   - WCAG/Standard: WCAG 1.1.1; responsive media best practice
   - Recommendation: Render artwork with semantic image elements or Nuxt image primitives, provide meaningful alt text where needed, and lazy-load non-critical assets.
   - Suggested command: `/harden` and `/optimize`

3. Theming is mostly hard-coded and effectively single-theme.
   - Location: `assets/css/main.css:4`, `components/WaitingRoomPanel.vue:54`, `components/OpenRoomListPanel.vue:61`, `pages/play.vue:219`, `components/gameplay/GameplaySidebar.vue:27`, `components/gameplay/GameplayPlayer.vue:49`
   - Severity: High
   - Category: Theming
   - Description: The app defines a few root variables, but most surfaces still use raw `rgba(...)`, `bg-white/...`, and one-off amber values directly in templates.
   - Impact: Theme changes are expensive, dark mode is effectively unsupported, and contrast tuning must be repeated manually across components.
   - WCAG/Standard: design token and adaptive-theme best practice
   - Recommendation: Move color, border, shadow, and overlay decisions into a token system and map components to those tokens instead of inline utility literals.
   - Suggested command: `/normalize`

4. Development-status copy is exposed in core product surfaces.
   - Location: `components/AppHeader.vue:9`, `pages/play.vue:221`, `components/gameplay/GameplaySidebar.vue:79`, `components/WaitingRoomPanel.vue:127`
   - Severity: High
   - Category: Anti-Patterns / UX Writing
   - Description: Several surfaces explain that the app is mid-migration, HTTP-polled, or still missing later card-rule phases.
   - Impact: The interface spends high-value space on implementation details instead of helping players understand what to do next, which weakens trust and makes the product feel unfinished.
   - WCAG/Standard: frontend-design anti-patterns; UX writing best practice
   - Recommendation: Remove engineering-status prose from player-facing surfaces or move it behind a developer/debug context.
   - Suggested command: `/clarify`

### Medium-Severity Issues

1. The gameplay page introduces a second top-level heading beneath a persistent site `h1`.
   - Location: `components/AppHeader.vue:7`, `pages/play.vue:220`
   - Severity: Medium
   - Category: Accessibility
   - Description: `AppHeader` always renders `Stupefy` as an `h1`, and the gameplay page adds another `h1` for the room name.
   - Impact: Screen-reader heading navigation becomes less predictable, and document structure is weaker than necessary.
   - WCAG/Standard: WCAG 1.3.1
   - Recommendation: Reserve one page-level `h1` and demote repeated branding headings on interior pages.
   - Suggested command: `/normalize`

2. The gameplay metrics strip reads like a template dashboard and weakens information hierarchy.
   - Location: `pages/play.vue:272`
   - Severity: Medium
   - Category: Anti-Patterns / Responsive
   - Description: The five-card row of `Player`, `Phase`, `Targets`, `Room Key`, and `Last Sync` matches the "hero metrics/card grid" pattern called out in the frontend-design skill.
   - Impact: Important game controls compete with low-value telemetry, and the layout becomes denser on smaller screens without clearly improving play.
   - WCAG/Standard: frontend-design anti-patterns
   - Recommendation: Collapse secondary telemetry into a compact status bar or sidebar detail area and prioritize gameplay state over implementation metadata.
   - Suggested command: `/distill`

3. Fixed card dimensions and dense wrapping make the gameplay surface fragile at narrow widths and high zoom.
   - Location: `components/gameplay/GameplayCard.vue:27`, `components/gameplay/GameplayCharacter.vue:28`, `components/gameplay/GameplayPlayer.vue:72`
   - Severity: Medium
   - Category: Responsive
   - Description: Cards and character panels use fixed width and height pairs, then rely on `flex-wrap` for overflow control.
   - Impact: The board will become visually crowded on smaller laptops, mobile widths, and 200% text zoom, even when it technically avoids horizontal scrolling.
   - WCAG/Standard: WCAG 1.4.10, responsive design best practice
   - Recommendation: Introduce component-level responsive variants, container-aware sizing, and alternate compact states for high-density board contexts.
   - Suggested command: `/normalize`

4. Waiting-room state refresh uses fixed polling rather than adaptive updates.
   - Location: `pages/waiting-room.vue:167`
   - Severity: Medium
   - Category: Performance
   - Description: The waiting room refreshes the whole room every 4 seconds regardless of visibility or user activity.
   - Impact: This increases unnecessary network activity and can feel sluggish compared with event-driven updates.
   - WCAG/Standard: performance best practice
   - Recommendation: Pause polling when the tab is hidden, back off on idle rooms, or move more of the room state to realtime updates.
   - Suggested command: `/optimize`

### Low-Severity Issues

1. Typography falls back to generic platform serif faces.
   - Location: `assets/css/main.css:16`
   - Severity: Low
   - Category: Anti-Patterns / Theming
   - Description: The app uses `Georgia` and `Times New Roman` rather than a deliberate display/body pairing.
   - Impact: The visual identity feels less intentional than the art direction suggests.
   - WCAG/Standard: frontend-design typography guidance
   - Recommendation: Define a purposeful type system with a distinctive display face and a readable body face.
   - Suggested command: `/frontend-design`

2. Frosted-glass panel styling is overused as the default surface treatment.
   - Location: `assets/css/main.css:31`
   - Severity: Low
   - Category: Anti-Patterns
   - Description: `backdrop-filter: blur(14px)` is applied to the shared `.stu-panel` class, which makes most surfaces visually similar.
   - Impact: Hierarchy flattens and the interface starts to resemble a generic "glass card" composition.
   - WCAG/Standard: frontend-design anti-patterns
   - Recommendation: Reserve blur for rare emphasis surfaces and differentiate primary, secondary, and utilitarian containers.
   - Suggested command: `/distill`

3. Empty states are functional but not instructive.
   - Location: `components/OpenRoomListPanel.vue:78`, `components/WaitingRoomPanel.vue:149`
   - Severity: Low
   - Category: UX Writing / Onboarding
   - Description: Empty states explain that nothing is present, but they do not teach the next-best action or reduce uncertainty.
   - Impact: First-time users get less guidance than they could during quiet or empty moments.
   - WCAG/Standard: interaction and onboarding best practice
   - Recommendation: Add next steps, expectations, or recovery actions to empty states.
   - Suggested command: `/onboard`

## Patterns & Systemic Issues

- Hard-coded color literals appear across the app instead of design tokens, making theming inconsistent and expensive to maintain.
- Engineering-status copy appears in multiple player-facing surfaces, which systematically lowers perceived polish.
- Image-heavy gameplay controls are implemented as styled buttons with background images, combining accessibility and performance debt in one pattern.
- Rounded frosted cards are the default answer for most layout problems, so hierarchy depends on repetition rather than strong composition.

## Positive Findings

- Room create/join forms use labeled fields, which is a strong baseline for form accessibility.
- Most custom interactive list items are real buttons rather than click-only `div` elements.
- The lobby and waiting-room layouts already use single-column fallbacks before introducing wider two-column compositions.
- Nuxt UI color aliases are defined in `app.config.ts`, so there is at least a starting point for a proper tokenized theme.

## Recommendations by Priority

1. Immediate
   - Fix the unlabeled waiting-room chat controls.
   - Replace custom gameplay overlays with accessible dialog primitives.

2. Short-term
   - Remove migration and transport-status copy from player-facing surfaces.
   - Normalize colors, borders, shadows, and overlays into reusable tokens.
   - Rework card and character media rendering for semantic images and lazy loading.

3. Medium-term
   - Redesign the gameplay status strip to reduce template-dashboard feel.
   - Add denser responsive board variants for narrow screens and zoomed layouts.
   - Reduce polling cost with visibility-aware or realtime refresh logic.

4. Long-term
   - Establish a more intentional typography system.
   - Differentiate surface treatments so blur is used selectively rather than globally.
   - Improve empty states to teach room creation, room joining, and waiting-room behavior.

## Suggested Commands for Fixes

- Use `/harden` to address the modal accessibility issues, chat labeling issues, and background-image accessibility debt.
- Use `/normalize` to replace hard-coded colors with tokens and clean up heading hierarchy and responsive component variants.
- Use `/clarify` to remove engineering-status prose and rewrite player-facing guidance.
- Use `/optimize` to improve waiting-room refresh behavior and media loading.
- Use `/distill` to reduce the dashboard/card-template feel on the gameplay page.
- Use `/onboard` to improve empty states and first-time room flow guidance.
